# Отчёт по реальной выгрузке Инфо-Предприятия

Файлы, проверенные на живой выгрузке:
- `import.xml`
- `offers.xml`
- каталог изображений `import_files`

## Что обнаружено

- Формат: CommerceML 2.10
- Пространство имён: `urn:1C.ru:commerceml_2`
- Товаров в каталоге: 3510
- Категорий: 276
- Типов цен: 1 (`Мелкоооптовая`)
- Предложений в `offers.xml`: 3510
- Товаров с ненулевым остатком: 3017
- Переданных изображений: 10 файлов

## Особенности именно этой выгрузки

1. У товаров **нет `Артикул`**, поэтому основным идентификатором по факту выступает `Ид`.
2. Привязка товара к категории приходит не через `<Группа>`, а через:
   ```xml
   <Группы>
       <Ид>06640</Ид>
   </Группы>
   ```
3. Тег `<Картинка />` в `import.xml` пустой, поэтому прямой путь к картинке из XML не используется.
4. Реальные изображения лежат в `import_files` и совпадают по имени файла с `Ид` товара, например `02022.jpg`.
5. В `offers.xml` остаток приходит общим полем `<Количество>`, без детализации по складам.
6. Цена приходит одной категорией `Мелкоооптовая`.

## Что было доработано в проекте

- импорт категорий переписан под схему `<Группы><Ид>`;
- товар теперь корректно получает категорию по реальному XML;
- картинки подхватываются fallback-логикой по имени файла из `import_files`, даже если `<Картинка />` пустой;
- экспорт `orders.xml` переведён на namespaced CommerceML 2.10;
- проект проверен на живой выгрузке: импорт проходит без ошибок.

## Что ещё нужно для боевого двустороннего обмена

Чтобы довести интеграцию до конца, теперь нужны реальные примеры:
- `orders.xml`, который сайт должен отдавать обратно в Инфо-Предприятие;
- файл или XML со статусами заказов, если статусы возвращаются отдельной выгрузкой.

Без них мы уже закрыли **реальный импорт каталога/цен/остатков/картинок**, но ещё не можем финально пришить **боевой формат обратного заказа и статусов**.
